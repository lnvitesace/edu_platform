## 第二部分:核心技术详解

### 2.1 开篇:为什么需要JWT?

想象你去游乐园玩。有两种方式可以证明你买了门票:

**方式1:传统方式(Session)**
- 买票后,工作人员在登记本上写下你的名字
- 给你一个号码牌"1234"
- 每次玩项目,你出示号码牌,工作人员翻登记本确认你买过票
- **问题**:游乐园有100个项目,每个项目都要翻登记本,效率低

**方式2:现代方式(JWT)**
- 买票后,工作人员给你一张**防伪手环**
- 手环上印着:你的名字、有效期、游乐园印章(防伪标记)
- 每次玩项目,工作人员看一眼手环就知道你买过票
- **优点**:不需要翻登记本,工作人员只需验证印章是否正品

JWT就是这个**防伪手环**。

### 2.2 JWT到底是什么?

JWT全称是**JSON Web Token**,本质是一个**加密字符串**。

#### 长什么样?

```
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJ1c2VySWQiOiIxMjM0NSIsInVzZXJuYW1lIjoiam9obiJ9.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c
```

看起来像乱码?其实它分为**三段**,用`.`分隔:

```
第一段(头部).第二段(数据).第三段(签名)
```

#### 三段分别是什么?

**第一段:头部(Header)** - 说明书

```json
{
  "alg": "HS256",  // 用什么加密算法
  "typ": "JWT"     // 这是JWT类型
}
```

**第二段:数据(Payload)** - 真正的内容

```json
{
  "userId": "12345",        // 用户ID
  "username": "john",       // 用户名
  "role": "STUDENT",        // 角色
  "exp": 1735712400         // 过期时间(时间戳)
}
```

**第三段:签名(Signature)** - 防伪印章

```
把"第一段.第二段"用密钥加密 → 生成签名
```

**关键点**:只有服务器知道密钥,黑客无法伪造签名!

---

### 2.3 为什么JWT是安全的?

**场景:黑客想伪造管理员权限**

```
原始JWT:
第二段数据: {"role": "STUDENT"}
签名: abc123 (用密钥X计算得出)

黑客修改:
第二段数据: {"role": "ADMIN"}  ← 想提权
签名: xyz789 (随便写的)

服务器验证:
1. 解析JWT
2. 用密钥X重新计算签名
3. 计算结果: def456
4. 比较: xyz789 ≠ def456
5. 结论: 签名不匹配 → 拒绝请求!
```

**为什么黑客无法成功?**

因为黑客不知道密钥X!即使修改了数据,也无法生成正确的签名。

**类比**:
- JWT = 人民币
- 签名 = 防伪水印
- 密钥 = 造币厂的秘密技术
- 黑客 = 假币制造者(没有技术,造不出真钞)

### 2.4 双令牌机制:一长一短的智慧

**问题**:JWT有效期设多久?

```
设1小时:
  好处 → 被盗后1小时失效,安全
  坏处 → 用户每小时要重新登录,烦人

设7天:
  好处 → 用户不用频繁登录,方便
  坏处 → 被盗后7天都能用,危险
```

**解决方案**:发两个令牌!

```
访问令牌(Access Token):
  - 有效期: 1小时
  - 用途: 访问所有API
  - 存哪: 浏览器内存/LocalStorage

刷新令牌(Refresh Token):
  - 有效期: 7天
  - 用途: 获取新的访问令牌
  - 存哪: 数据库(可以随时撤销)
```

**工作流程**:

```
用户登录
  ↓
服务器发两个令牌
  ↓
用户用访问令牌调用API(1小时内正常使用)
  ↓
1小时后,访问令牌过期
  ↓
前端自动用刷新令牌换新的访问令牌
  ↓
用户无感知,继续使用
  ↓
7天后,刷新令牌也过期
  ↓
用户需要重新登录
```

**安全性**:
- 访问令牌被盗 → 1小时后失效
- 刷新令牌被盗 → 管理员可以从数据库删除,立即失效

### 2.5 Spring Security过滤链:层层安检

#### 什么是过滤器?

**类比:机场安检**

```
乘客(HTTP请求)进入机场:

第1道安检:检查身份证
  → 有身份证 → 通过
  → 没有 → 拒绝登机

第2道安检:行李扫描
  → 无危险品 → 通过
  → 有刀具 → 拒绝登机

第3道安检:人脸识别
  → 不是通缉犯 → 通过
  → 是通缉犯 → 抓起来

通过所有安检 → 可以登机
```

Spring Security的过滤器就是这些**安检关卡**。

#### 我们的过滤链(简化版)

```
HTTP请求进来
  ↓
第1关: CorsFilter
  → 检查请求来源(前端域名是否在白名单)
  ↓
第2关: JwtAuthenticationFilter (我们写的!)
  → 检查JWT令牌
  → 验证签名
  → 解析用户信息
  ↓
第3关: AuthorizationFilter
  → 检查用户权限
  → /api/auth/** → 所有人可访问
  → /api/admin/** → 只有管理员可访问
  ↓
通过所有关卡 → 执行业务代码
```

#### JwtAuthenticationFilter做什么?

**核心逻辑**(伪代码):

```
每个请求进来:

1. 从请求头提取JWT
   Authorization: Bearer eyJhbGci...
   → 去掉"Bearer ",得到JWT

2. 验证JWT
   → 签名正确吗? ✓
   → 过期了吗? ✗

3. 解析用户ID
   → userId = 12345

4. 从数据库加载用户完整信息
   → 为什么不直接用JWT里的信息?
   → 因为要确保用户没被禁用/删除

5. 告诉Spring Security:当前用户是谁
   → SecurityContextHolder.setContext(用户信息)

6. 放行请求,继续下一个过滤器
```

**关键代码**(只看核心部分):

```java
// 从请求头提取JWT
String jwt = request.getHeader("Authorization").substring(7);

// 验证JWT
if (tokenProvider.validateToken(jwt)) {
    // 解析用户ID
    Long userId = tokenProvider.getUserIdFromToken(jwt);

    // 加载用户
    UserDetails user = userDetailsService.loadUserById(userId);

    // 设置到安全上下文
    SecurityContextHolder.getContext().setAuthentication(user);
}

// 继续下一个过滤器
filterChain.doFilter(request, response);
```

### 2.6 SecurityConfig:安全系统的总控制室

这个类决定了:
1. 哪些URL需要登录才能访问
2. 哪些角色可以访问哪些接口
3. 密码怎么加密
4. CORS怎么配置

**核心配置**(简化版):

```java
@Configuration
public class SecurityConfig {

    // 1. 密码加密器
    @Bean
    public PasswordEncoder passwordEncoder() {
        return new BCryptPasswordEncoder();
        // BCrypt:目前最安全的密码加密算法
    }

    // 2. 安全规则
    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) {
        http
            // 禁用CSRF(因为用JWT,不用Cookie)
            .csrf(csrf -> csrf.disable())

            // 启用CORS(允许前端跨域访问)
            .cors(cors -> cors.configurationSource(corsConfig()))

            // 无状态会话(不用Session)
            .sessionManagement(session ->
                session.sessionCreationPolicy(STATELESS)
            )

            // URL访问规则
            .authorizeHttpRequests(authz -> authz
                .requestMatchers("/api/auth/**").permitAll()      // 所有人可访问
                .requestMatchers("/api/admin/**").hasRole("ADMIN") // 只有管理员
                .anyRequest().authenticated()                      // 其他需要登录
            )

            // 添加JWT过滤器
            .addFilterBefore(jwtFilter, UsernamePasswordAuthenticationFilter.class);

        return http.build();
    }

    // 3. CORS配置
    @Bean
    public CorsConfigurationSource corsConfig() {
        CorsConfiguration config = new CorsConfiguration();

        // 允许的前端地址
        config.setAllowedOrigins(List.of(
            "http://localhost:3000",  // React开发服务器
            "http://localhost:5173"   // Vite开发服务器
        ));

        // 允许的HTTP方法
        config.setAllowedMethods(List.of("GET", "POST", "PUT", "DELETE"));

        // 允许的请求头
        config.setAllowedHeaders(List.of("Authorization", "Content-Type"));

        return source;
    }
}
```

### 2.7 常见问题

**Q1: 为什么要禁用CSRF?**

```
传统Web应用:
  登录后 → 服务器设置Cookie
  后续请求 → 浏览器自动带上Cookie
  问题 → 恶意网站可以诱导用户点击,浏览器自动带上Cookie发请求

我们的应用:
  登录后 → JWT存在LocalStorage
  后续请求 → 前端手动添加Authorization头
  安全 → 恶意网站读不到LocalStorage,无法伪造请求
```

**Q2: 为什么用无状态会话?**

```
有状态(Session):
  服务器A存了用户会话
  用户下次请求到服务器B
  问题 → 服务器B找不到会话 → 需要Session共享(复杂!)

无状态(JWT):
  用户每次请求都带JWT
  任何服务器都能验证JWT(只需要密钥)
  优点 → 容易扩展,服务器不存储状态
```

**Q3: JWT被盗怎么办?**

```
多层防护:
1. 短有效期 → 访问令牌1小时失效
2. HTTPS传输 → 防止中间人窃取
3. 刷新令牌可撤销 → 发现被盗立即删除数据库记录
4. 黑名单(可选) → 将被盗令牌加入黑名单
```

## 总结

**核心概念**:

| 概念 | 生活类比 | 技术实现 |
|------|---------|----------|
| JWT | 游乐园防伪手环 | 包含用户信息的加密字符串 |
| 签名 | 人民币防伪水印 | 用密钥加密,防止篡改 |
| 过滤器 | 机场安检关卡 | 层层验证请求 |
| 访问令牌 | 1小时有效的临时通行证 | 1小时过期的JWT |
| 刷新令牌 | 7天有效的续签凭证 | 7天过期,可撤销 |

**记住这一点**:

JWT的核心思想是**把用户信息放在令牌里**,这样服务器不用存储会话,只需要验证令牌的签名。就像防伪手环,工作人员不需要查登记本,看一眼手环就知道你是否买过票。
