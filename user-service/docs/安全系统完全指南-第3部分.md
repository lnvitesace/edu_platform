# 教育平台后端 - 安全系统完全指南（第3部分）

> 本文档用故事化的方式讲解登录、登出、令牌刷新的完整流程,假设你从未学过计算机安全

---

## 第三部分:真实场景下的完整流程

### 3.1 故事开始:小明的一天

**背景**:
- 小明是一名学生,想用教育平台学习
- 教育平台分为前端(网页)和后端(服务器)
- 小明需要先注册账号,然后登录

---

### 3.2 场景1:小明注册账号

#### 用户视角

```
小明打开网站 → 点击"注册"
填写信息:
  用户名: ming_student
  邮箱: ming@example.com
  密码: MyPass123
  姓名: 小明
  角色: 学生

点击"提交" → 等待几秒 → 注册成功!
自动跳转到首页,已经登录了
```

#### 幕后发生了什么?

**第1步:前端发送注册请求**

```
前端(浏览器)发送请求到服务器:

POST /api/auth/register
{
  "username": "ming_student",
  "email": "ming@example.com",
  "password": "MyPass123",
  "firstName": "小",
  "lastName": "明",
  "role": "STUDENT"
}
```

**第2步:服务器验证数据**

```
服务器检查:
  用户名长度: 3-20字符 ✓
  邮箱格式: 正确 ✓
  密码强度: 至少8位,有大小写和数字 ✓
  用户名是否已存在: 查数据库 → 不存在 ✓
  邮箱是否已存在: 查数据库 → 不存在 ✓

全部通过 → 继续
```

**第3步:加密密码**

```
原始密码: MyPass123

服务器:
  "密码不能明文存储,太危险了!"
  "用BCrypt加密"

加密过程:
  1. 生成随机盐(salt): a7s8d9f0...
  2. 用盐混合密码: MyPass123 + a7s8d9f0...
  3. 多次哈希运算(4096次)
  4. 得到密码哈希: $2a$10$N9qo8uLO...

存储到数据库: $2a$10$N9qo8uLO...
```

**为什么要加密?**

```
假设黑客入侵了数据库:

不加密:
  数据库存的是 MyPass123
  黑客直接看到密码 → 可以登录小明的账号 💀

加密:
  数据库存的是 $2a$10$N9qo8uLO...
  黑客看到的是乱码 → 无法反推出原始密码 ✓
  而且每个用户的盐不同,即使两人密码相同,哈希值也不同
```

**第4步:保存用户信息**

```
服务器保存两张表:

users表:
  id: 1
  username: ming_student
  email: ming@example.com
  password: $2a$10$N9qo8uLO... (加密后)
  role: STUDENT
  status: ACTIVE

user_profiles表:
  user_id: 1
  first_name: 小
  last_name: 明
  avatar_url: null
  bio: null
```

**第5步:生成JWT令牌**

```
服务器:
  "注册成功,给你两个通行证"

生成访问令牌(1小时有效):
  头部: {"alg": "HS256", "typ": "JWT"}
  数据: {"userId": 1, "username": "ming_student", "role": "STUDENT"}
  签名: 用密钥加密

生成刷新令牌(7天有效):
  只包含: {"userId": 1}
  也存到数据库的refresh_tokens表
```

**第6步:返回给前端**

```
服务器响应:

{
  "accessToken": "eyJhbGci...",
  "refreshToken": "eyJ1c2Vy...",
  "expiresIn": 3600000,  // 1小时(毫秒)
  "user": {
    "id": 1,
    "username": "ming_student",
    "email": "ming@example.com",
    "role": "STUDENT",
    "firstName": "小",
    "lastName": "明"
  }
}
```

**第7步:前端保存令牌**

```
前端收到响应:
  "太好了,注册成功!"
  把accessToken存到LocalStorage
  把refreshToken也存到LocalStorage
  显示用户信息
  跳转到首页
```

**完整流程图**:

```
小明填表单 → 前端发送请求
                 ↓
              服务器验证
                 ↓
           密码加密(BCrypt)
                 ↓
          保存到数据库(2张表)
                 ↓
           生成JWT令牌(2个)
                 ↓
              返回给前端
                 ↓
          前端保存到LocalStorage
                 ↓
              显示首页
```

### 3.3 场景2:小明登录(第二天)

#### 用户视角

```
小明第二天打开网站 → 发现退出登录了(刷新令牌过期或清空了缓存)
点击"登录"
输入:
  用户名或邮箱: ming_student
  密码: MyPass123

点击"登录" → 登录成功!
```

#### 幕后发生了什么?

**第1步:前端发送登录请求**

```
POST /api/auth/login
{
  "usernameOrEmail": "ming_student",
  "password": "MyPass123"
}
```

**第2步:服务器验证密码**

```
服务器:
  "先从数据库找这个用户"

查询数据库:
  SELECT * FROM users
  WHERE username = 'ming_student' OR email = 'ming_student'

找到了:
  id: 1
  username: ming_student
  password: $2a$10$N9qo8uLO... (数据库里的密码哈希)
  role: STUDENT
  status: ACTIVE

验证密码:
  1. 用户输入: MyPass123
  2. 数据库存的: $2a$10$N9qo8uLO...
  3. BCrypt验证过程:
     - 从哈希值提取盐: a7s8d9f0...
     - 用同样的盐加密输入密码
     - 比较结果和数据库哈希值
     - 一致 → 密码正确 ✓
```

**为什么BCrypt安全?**

```
黑客尝试暴力破解:

普通MD5哈希:
  破解速度: 每秒10亿次
  8位密码: 几小时就能破解 💀

BCrypt(4096次迭代):
  破解速度: 每秒几千次
  8位密码: 需要几十年 ✓

而且BCrypt可以调整成本因子,计算机性能提升后可以增加迭代次数
```

**第3步:检查用户状态**

```
服务器:
  "密码正确,但还要检查账号状态"

检查:
  status == ACTIVE ✓
  (如果是SUSPENDED或DELETED,拒绝登录)

通过 → 继续
```

**第4步:生成新令牌**

```
生成新的访问令牌和刷新令牌

删除旧刷新令牌:
  DELETE FROM refresh_tokens WHERE user_id = 1

保存新刷新令牌:
  INSERT INTO refresh_tokens (token, user_id, expiry_date)
  VALUES ('eyJ1c2Vy...', 1, '2025-01-09')

为什么删除旧的?
  - 防止令牌泄露后被滥用
  - 一个用户同时只能有一个有效刷新令牌
```

**第5步:返回令牌**

```
{
  "accessToken": "eyJhbGci...",
  "refreshToken": "eyJ1c2Vy...",
  "expiresIn": 3600000,
  "user": { ... }
}

前端保存到LocalStorage → 登录成功
```

### 3.4 场景3:小明访问课程页面

#### 用户视角

```
小明登录后,点击"我的课程"
页面立即显示课程列表
```

#### 幕后发生了什么?

**第1步:前端发送请求**

```
GET /api/users/me

前端自动添加令牌:
  Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
```

**第2步:请求经过过滤器链**

```
请求到达服务器 → 经过多个过滤器:

【第1关:CorsFilter】
  检查请求来源: http://localhost:3000
  是否在白名单? 是 ✓
  → 通过

【第2关:JwtAuthenticationFilter】
  1. 提取Authorization头
     "Bearer eyJhbGci..."
     → 去掉"Bearer ",得到JWT

  2. 验证JWT签名
     用服务器密钥重新计算签名
     与JWT中的签名比较
     → 一致 ✓

  3. 检查是否过期
     当前时间: 2025-01-02 10:30
     JWT过期时间: 2025-01-02 11:00
     → 未过期 ✓

  4. 解析用户ID
     从JWT的payload提取: userId = 1

  5. 加载用户完整信息
     从数据库查询user_id = 1
     得到用户对象(包括权限)

  6. 设置到安全上下文
     SecurityContextHolder.setContext(用户1)
     → 现在Spring Security知道当前用户是谁了

  → 通过

【第3关:AuthorizationFilter】
  检查URL权限:
    请求的URL: /api/users/me
    规则: /api/users/** → 需要登录
    当前用户: 已登录 ✓
  → 通过

所有关卡通过 → 请求到达Controller
```

**第3步:Controller处理请求**

```java
@GetMapping("/me")
public UserResponse getCurrentUser(
    @AuthenticationPrincipal UserPrincipal userPrincipal
) {
    // @AuthenticationPrincipal自动注入当前用户
    // 从SecurityContextHolder中获取

    Long userId = userPrincipal.getId();  // 1
    return userService.getCurrentUser(userId);
}
```

**第4步:返回用户信息**

```
{
  "id": 1,
  "username": "ming_student",
  "email": "ming@example.com",
  "role": "STUDENT",
  "firstName": "小",
  "lastName": "明",
  "avatarUrl": null,
  "bio": null
}

前端显示在页面上
```

### 3.5 场景4:1小时后,访问令牌过期了

#### 用户视角

```
小明在写作业,写了1个小时
点击"保存"
→ 没有弹出"请重新登录",作业保存成功!
(用户完全没感知到令牌过期)
```

#### 幕后发生了什么?

**第1步:前端发送请求**

```
POST /api/assignments/submit

前端添加访问令牌:
  Authorization: Bearer eyJhbGci... (已过期)
```

**第2步:服务器拒绝请求**

```
JwtAuthenticationFilter验证:
  签名: 正确 ✓
  过期时间: 2025-01-02 11:00
  当前时间: 2025-01-02 11:05
  → 已过期 ✗

返回: 401 Unauthorized
```

**第3步:前端自动刷新令牌**

```javascript
// 前端的axios拦截器
axios.interceptors.response.use(
  response => response,
  async error => {
    if (error.response.status === 401) {
      // 401错误,令牌可能过期了
      // 尝试用刷新令牌换新的访问令牌

      const refreshToken = localStorage.getItem('refreshToken');

      // 发送刷新请求
      const response = await axios.post('/api/auth/refresh', {
        refreshToken: refreshToken
      });

      // 保存新令牌
      const newAccessToken = response.data.accessToken;
      localStorage.setItem('accessToken', newAccessToken);

      // 重试原始请求
      error.config.headers.Authorization = `Bearer ${newAccessToken}`;
      return axios(error.config);
    }
  }
);
```

**第4步:服务器处理刷新请求**

```
POST /api/auth/refresh
{
  "refreshToken": "eyJ1c2Vy..."
}

服务器:
  1. 查询数据库
     SELECT * FROM refresh_tokens WHERE token = 'eyJ1c2Vy...'
     找到了 ✓

  2. 检查是否过期
     expiry_date: 2025-01-09
     当前时间: 2025-01-02
     → 未过期 ✓

  3. 获取关联的用户
     user_id: 1
     → 用户小明

  4. 生成新令牌
     新访问令牌: eyJhbGNpNew...
     新刷新令牌: eyJ1c2VyNew...

  5. 更新数据库
     UPDATE refresh_tokens
     SET token = 'eyJ1c2VyNew...', expiry_date = '2025-01-09'
     WHERE user_id = 1

  6. 返回新令牌
     {
       "accessToken": "eyJhbGNpNew...",
       "refreshToken": "eyJ1c2VyNew...",
       "expiresIn": 3600000
     }
```

**第5步:前端重试原始请求**

```
用新的访问令牌重新发送:
  POST /api/assignments/submit
  Authorization: Bearer eyJhbGNpNew...

这次成功 ✓
→ 作业保存成功

用户完全没感觉到刷新过程!
```

**整个过程**:

```
小明点击"保存"
  ↓
前端发送请求(旧令牌,已过期)
  ↓
服务器返回401
  ↓
前端拦截器检测到401
  ↓
自动用刷新令牌换新令牌(用户无感知)
  ↓
重新发送请求(新令牌)
  ↓
成功保存
  ↓
小明看到:"保存成功!"
```

### 3.6 场景5:小明登出

#### 用户视角

```
小明做完作业,点击"退出登录"
→ 跳转到登录页面
```

#### 幕后发生了什么?

**第1步:前端发送登出请求**

```
POST /api/auth/logout
Authorization: Bearer eyJhbGci...
```

**第2步:服务器处理**

```
服务器:
  "知道了,我帮你清理"

1. 从JWT解析用户ID
   userId = 1

2. 删除刷新令牌
   DELETE FROM refresh_tokens WHERE user_id = 1
   → 刷新令牌失效

3. 清除Redis缓存(如果有)
   DELETE user:session:1
```

**第3步:前端清理**

```javascript
// 删除本地存储的令牌
localStorage.removeItem('accessToken');
localStorage.removeItem('refreshToken');

// 跳转到登录页
window.location.href = '/login';
```

**为什么还要删除刷新令牌?**

```
只删除前端令牌的问题:
  黑客偷了刷新令牌(从备份或其他地方)
  → 可以继续生成访问令牌
  → 小明已经"登出",但黑客还能用账号

删除数据库的刷新令牌:
  黑客即使有刷新令牌
  → 服务器查数据库发现不存在
  → 拒绝刷新请求
  → 黑客无法继续使用
```

### 3.7 场景6:黑客尝试攻击(多种失败案例)

#### 攻击1:修改JWT提升权限

```
黑客拦截小明的JWT:
  原始: {"userId": 1, "role": "STUDENT"}

黑客修改:
  修改后: {"userId": 1, "role": "ADMIN"}

黑客发送请求:
  Authorization: Bearer 修改后的JWT

服务器验证:
  用密钥重新计算签名
  → 签名不匹配!
  → 拒绝请求

结果: 失败 ✗
```

#### 攻击2:暴力破解密码

```
黑客尝试:
  for password in常用密码表:
      尝试登录(username, password)

服务器(安装了速率限制):
  检测到同一IP,1分钟内登录失败5次
  → 封禁IP 10分钟
  → 返回429 Too Many Requests

黑客:
  "才试了5次就被封了,10亿个密码要试多久?"
  "放弃!"

结果: 失败 ✗
```

#### 攻击3:SQL注入

```
黑客尝试:
  用户名输入: admin' OR '1'='1
  希望SQL变成:
    SELECT * FROM users WHERE username = 'admin' OR '1'='1'
    → 这样会返回所有用户

实际情况:
  我们用JPA参数化查询:
    @Query("SELECT u FROM User u WHERE u.username = :username")
    → JPA自动转义特殊字符
    → 查询的是字面量 "admin' OR '1'='1"
    → 找不到这个用户名

结果: 失败 ✗
```

### 3.9 关键概念回顾

**1. 密码加密(BCrypt)**
- 类比:把鸡蛋打碎搅拌,无法还原
- 作用:即使数据库泄露,黑客也拿不到原始密码
- 特点:加盐、多次哈希、不可逆

**2. JWT令牌**
- 类比:防伪手环
- 作用:证明身份,避免频繁查数据库
- 安全:签名防篡改,过期时间限制风险

**3. 双令牌机制**
- 访问令牌:短期(1小时),常用
- 刷新令牌:长期(7天),换新令牌用,可撤销

**4. 过滤器链**
- 类比:机场多道安检
- 作用:层层验证请求合法性
- 顺序:CORS → JWT验证 → 权限检查 → 业务逻辑

**5. 无状态设计**
- 服务器不存储会话
- 所有信息在JWT中
- 好处:容易扩展到多台服务器

---

### 3.10 安全检查清单

开发完成后,确认以下几点:

- [ ] 密码用BCrypt加密存储(不是明文)
- [ ] JWT密钥足够长(至少32字节)
- [ ] 访问令牌有效期短(1小时)
- [ ] 刷新令牌存数据库,可撤销
- [ ] 敏感接口需要登录(不是permitAll)
- [ ] 管理接口检查角色(hasRole)
- [ ] CORS白名单配置正确(不是允许所有来源)
- [ ] 生产环境用HTTPS(不是HTTP)
- [ ] 输入数据验证(@Valid)
- [ ] 防止SQL注入(用JPA或参数化查询)
- [ ] 速率限制(防止暴力破解)
- [ ] 日志不记录密码等敏感信息

## 结语

现在你应该理解了:

1. **注册**:数据验证 → 密码加密 → 存数据库 → 生成JWT
2. **登录**:验证密码(BCrypt) → 生成JWT → 删除旧刷新令牌
3. **访问API**:提取JWT → 验证签名和过期时间 → 加载用户 → 检查权限
4. **刷新令牌**:查数据库 → 检查过期 → 生成新JWT → 更新数据库
5. **登出**:删除数据库刷新令牌 → 清除本地存储

整个系统的核心思想:**用密码学保证安全,用双令牌平衡安全性和用户体验,用过滤器链实现层层防护**。

记住,安全不是一次性的工作,而是持续的过程。定期检查、更新依赖、修复漏洞,才能让系统长期安全运行。
